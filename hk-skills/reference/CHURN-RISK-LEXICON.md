## 0. 이탈 위험도란? / What churn_risk means here

이 데모의 "이탈"은 장기 해지(attrition)가 아니라 **이번 통화 안에서 고객이 이탈(이탈 = 통화 중단·거절·무관심으로 빠져나감)할 확률**입니다.
관리자/상담원이 우상단 게이지로 *지금 개입이 필요한지*를 즉시 판단하기 위한 실시간 지표입니다.

- **0–33 (낮음, 초록)** — 고객이 듣고 있고 관심·진척 신호가 우세.
- **34–66 (보통, 노랑)** — 망설임/반론 혼재. AI Action으로 이탈 방지 필요.
- **67–100 (높음, 빨강)** — 종료/거절 신호 우세. 즉시 상담원 연결 또는 가치 재강조 필요.

> ⚠️ **주의 / Important nuance**: 본 제품에서 **한도조회·상담원 연결 요청은 성공 경로(S1 transfer)** 입니다
> (`PRODUCT-BRIEF.md` §3, `ARCHITECTURE.md` §3.2). 따라서 "상담원 바꿔줘"는 이탈 신호가 **아니라**
> 강한 **관심(이탈위험 ↓)** 신호로 취급합니다. 반대로 "관심 없어요/끊을게요"가 이탈(이탈위험 ↑) 신호입니다.

## 1. 점수 모델 / Scoring Model

키워드 가중치 합산 + EMA 평활(smoothing) 방식. 결정적(deterministic)이고 설명 가능해야 하므로
LLM 단독 판단이 아니라 **사전 기반 점수 + (선택) LLM 보정**으로 구성합니다.

```
baseline       = 50                      # 통화 시작 시 중립값
turn_raw(t)    = Σ weight(kw)            # 이번 턴 고객 발화에서 매칭된 모든 키워드 가중치 합
                  for kw matched in customer_utterance(t)
turn_score(t)  = clamp(baseline + turn_raw(t), 0, 100)
churn_risk(t)  = round( α · turn_score(t) + (1-α) · churn_risk(t-1) )   # EMA, α=0.6
churn_risk(t)  = clamp(churn_risk(t), 0, 100)
```

규칙 / Rules:

1. **방향 규약**: `weight > 0` = 이탈위험 **상승(cons)**, `weight < 0` = 이탈위험 **하락(pro)**.
2. **EMA 평활** (`α=0.6`): 한 번의 발화로 게이지가 튀지 않게 직전 값과 가중 평균. 첫 턴은 `churn_risk(0)=baseline`.
3. **턴 상한**: 한 턴의 `turn_raw`는 `[-40, +40]`로 clamp(폭주 방지). 동일 카테고리 중복 매칭은 **최대 2개**까지만 합산.
4. **부정 반전(negation)**: 키워드 앞 7자 이내에 `안·않·말고·없·아니·못` 등 부정어가 있으면 부호 반전
   (예: "관심 **없**어요" → pro 키워드 "관심"을 cons로 반전). §4 참조.
5. **강조 배수**: `진짜·완전·너무·전혀·절대·당장` 등 강조어가 키워드에 인접하면 가중치 ×1.5(소수 반올림).
6. **무발화/침묵**: 고객 응답이 STT 공백이거나 `네…`/`음…` 류 최소응답만 2턴 연속이면 +6(주저 신호).
7. **클램프**: 모든 단계에서 0–100.
8. **방출**: 갱신된 `churn_risk`는 AppSync 구독 `onIndexUpdate`로 push(DynamoDB write→Streams 팬아웃), `call(id)` 쿼리의
   `analysis.churn_risk`에 반영(`API.md` §1.2, §2.1).

> **LLM 보정(선택)**: `classify` 노드가 의미 기반으로 ±10 보정을 제안할 수 있으나, 사전 점수가 1차 진실이며
> 보정은 `rationale`에 근거를 남깁니다. 데모 안정성을 위해 사전 점수만으로도 동작해야 합니다.

## 2. CONS — 이탈위험 상승 사전 / Risk-RAISING dictionary (weight > 0)

> 고객이 빠져나가려는 신호. 매칭 시 `churn_risk` 상승.

### 2.1 통화 종료·거절 의사 (HANGUP_INTENT) — weight **+18**
가장 강한 즉시 이탈 신호.

| 표현 (한국어) | 비고 |
|---|---|
| 끊을게요 / 끊습니다 / 그만 끊죠 | 직접 종료 의사 |
| 됐어요 / 됐습니다 / 됐고요 | 강한 거절 |
| 관심 없어요 / 관심 없습니다 / 필요 없어요 | 관심 부정 |
| 안 할게요 / 안 합니다 / 가입 안 해요 | 가입 거절 |
| 전화 그만하세요 / 다시 전화하지 마세요 / 연락하지 마세요 | 재연락 거부 |
| 싫어요 / 싫습니다 | 명시적 거부 |
| 지금 안 받을게요 / 나중에 받을게요 | 통화 자체 회피 |

### 2.2 불신·사기 의심·짜증 (DISTRUST_ANGER) — weight **+14**
> 단, **금융사기 의심 발화**는 별도로 `detect_fraud` 노드가 `fraud_suspected` 플래그를 세웁니다
> (`ARCHITECTURE.md` §3.2). 여기서는 *이탈위험* 점수에만 기여하고 통화를 종료하지 않습니다.

| 표현 | 비고 |
|---|---|
| 보이스피싱 아니에요? / 사기 아니에요? / 사기 전화 | 사기 의심(→ fraud_suspected 연동) |
| 어떻게 알았어요 / 내 번호 어떻게 / 개인정보 어디서 | 정보출처 불신 |
| 못 믿겠어요 / 믿을 수 없어요 / 의심스럽네요 | 신뢰 부족 |
| 진짜 회사 맞아요? / 사칭 아니에요? | 사칭 의심 |
| 짜증나요 / 귀찮아요 / 그만 좀 / 왜 자꾸 | 짜증·피로 |
| 신고할게요 / 금감원에 / 고소 | 강한 적대 |

### 2.3 명시적 무관심·반복 거절 (NOT_INTERESTED) — weight **+12**

| 표현 | 비고 |
|---|---|
| 생각 없어요 / 생각 없습니다 | 의향 없음 |
| 필요할 때 연락할게요 / 제가 알아서 할게요 | 완곡 거절 |
| 이미 (대출) 받았어요 / 이미 있어요 | 수요 충족 |
| 다른 데서 하고 있어요 / 거래 은행 있어요 | 경쟁사 고정 |
| 관심 있으면 제가 신청하죠 | 수동적 거절 |
| 광고는 안 받아요 / 영업 전화 사절 | 영업 거부 |

### 2.4 가격·조건 반론 (PRICE_OBJECTION) — weight **+8**
이탈로 직결되진 않지만 이탈 위험을 높이는 망설임.

| 표현 | 비고 |
|---|---|
| 금리가 너무 높네요 / 비싸요 | 금리 불만 |
| 한도가 너무 적어요 / 한도가 작네요 | 한도 불만 |
| 조건이 별로네요 / 메리트가 없네요 | 조건 불만 |
| 수수료 있어요? / 중도상환 수수료 | 비용 우려(반론성) |
| 그 정도면 지금이랑 비슷한데 | 차별성 부족 |

### 2.5 시간 압박·회피 (TIME_PRESSURE) — weight **+7**

| 표현 | 비고 |
|---|---|
| 바빠요 / 지금 바빠서 / 회의 중 / 운전 중 | 즉시 회피 |
| 나중에요 / 다음에요 / 이따가 | 지연 회피 |
| 시간 없어요 / 빨리요 / 용건만 | 시간 압박 |
| 문자로 보내주세요 / 문자나 주세요 | 통화 회피(브러시오프) |
| 바쁘니까 끊고 (이어서) | 회피+종료 결합 |

### 2.6 주저·약한 신호 (HESITATION) — weight **+5**

| 표현 | 비고 |
|---|---|
| 글쎄요 / 음… 잘 모르겠는데 | 망설임 |
| 생각해 볼게요 / 고민해 볼게요 | 보류(약한 이탈) |
| 와이프랑 / 가족이랑 상의 / 남편이랑 | 결정 지연 |
| 잘 모르겠어요 / 어렵네요 | 이해/확신 부족 |
| 별로… / 글쎄… (말끝 흐림) | 소극 반응 |

## 3. PRO — 이탈위험 하락 사전 / Risk-LOWERING dictionary (weight < 0)

> 고객이 관심·진척을 보이는 신호. 매칭 시 `churn_risk` 하락.

### 3.1 한도조회·상담원 연결 요청 (TRANSFER_INTENT) — weight **−20**
> **성공 경로(S1)**. 본 제품에서 가장 강한 *긍정* 신호이며 이탈위험을 크게 낮춥니다
> (`PRODUCT-BRIEF.md` §3, `API.md` §1.2 `/transfer`).

| 표현 | 비고 |
|---|---|
| 한도 조회해 주세요 / 제 한도 얼마예요? / 한도 좀 알아봐 | 한도조회 요청 |
| 상담원 연결해 주세요 / 사람이랑 통화 / 담당자 바꿔줘 | 상담원 연결 요청 |
| 자세히 상담받고 싶어요 / 직접 상담 | 상담 요청 |
| 신청하려면 어떻게 / 진행해 주세요 / 접수해 주세요 | 진행 의사 |

### 3.2 구매·진척 의사 (BUYING_INTENT) — weight **−16**

| 표현 | 비고 |
|---|---|
| 할게요 / 하겠습니다 / 가입할게요 | 가입 의사 |
| 갈아탈게요 / 대환할게요 / 옮길게요 | 대환 의사 |
| 좋네요 / 좋은데요 / 괜찮네요 / 마음에 들어요 | 긍정 평가 |
| 그렇게 해주세요 / 그걸로 할게요 | 선택 확정 |
| 언제부터 가능해요? / 지금 바로 돼요? | 실행 시점 질문(강한 의사) |

### 3.3 정보 탐색·관심 질문 (INTEREST_QUESTION) — weight **−10**
질문은 평가 중이라는 관여 신호(HubSpot/Gong: "Does it do X?" = 관심).

| 표현 | 비고 |
|---|---|
| 금리가 어떻게 돼요? / 금리 얼마예요? | 금리 문의 |
| 한도가 얼마까지 / 얼마나 받을 수 있어요? | 한도 문의 |
| 조건이 어떻게 돼요? / 자격이 되나요? | 자격/조건 문의 |
| 기존 대출보다 나아요? / 얼마나 절약돼요? | 비교 문의(가치 탐색) |
| 서류 뭐 필요해요? / 어떻게 진행돼요? | 절차 문의 |
| 차 담보로도 돼요? / 자동차담보 | 상품 적합성 문의 |

### 3.4 경청·긍정 수용 (ENGAGEMENT) — weight **−6**

| 표현 | 비고 |
|---|---|
| 아 네 / 그렇군요 / 그래요? | 능동적 경청 |
| 더 말해보세요 / 설명해 주세요 / 계속하세요 | 추가 설명 요청 |
| 아 그게 그런 거예요? / 이해됐어요 | 이해·수용 |
| 맞아요 / 그렇죠 / 동의해요 | 공감 |
| (필요·고민) 있긴 한데 / 안 그래도 알아보던 중 | 잠재 니즈 노출 |

## 4. 부정·강조 처리 / Negation & Intensifiers

매칭 시 키워드 단독이 아니라 **인접 토큰**을 함께 봐야 오판을 막습니다.

| 케이스 | 예 | 처리 |
|---|---|---|
| 부정어 + pro 키워드 | "관심 **없**어요" | pro("관심")의 부호 반전 → cons. |
| 부정어 + cons 키워드 | "바쁜 건 **아니**고" | cons("바쁘") 무효화(가중치 0). |
| 강조어 + cons | "**전혀** 관심 없어요" | 가중치 ×1.5. |
| 강조어 + pro | "**진짜** 좋네요" | 가중치 ×1.5. |
| 이중부정 | "안 할 수가 **없**죠" | 부정 2회 → pro 유지. |

- 부정어 셋: `안 / 않 / 없 / 못 / 아니 / 말고 / 됐` (키워드 좌측 7자 윈도우).
- 강조어 셋: `진짜 / 완전 / 너무 / 전혀 / 절대 / 당장 / 정말 / 아주`.
- 매칭은 **정규화 후** 수행: 공백·조사 변이 흡수를 위해 핵심 어간 기준 부분 일치
  (예: "끊을게요/끊습니다/끊죠" → 어간 `끊`). `churn_risk_lexicon.json`은 `stems` 배열로 어간을 제공.

## 5. 동작 예시 / Worked Example (S1)

```
churn_risk(0) = 50
T1 고객: "여보세요? 누구세요"            → 매칭 없음. turn=50. EMA: 0.6·50+0.4·50 = 50
T2 고객: "아 관심 없어요, 바빠서"        → -없음반전- "관심없"(+12) + "바빠"(+7) = +19
        turn_raw=+19 → turn=69. EMA: 0.6·69+0.4·50 = 61  (노랑→빨강 경계)
T3 봇:  가치 재강조(월 4만원 절감)        → (봇 발화는 점수화 안 함)
T4 고객: "음… 금리가 어떻게 되는데요?"    → "금리어떻게"(-10) = -10
        turn_raw=-10 → turn=40. EMA: 0.6·40+0.4·61 = 48  (노랑, 회복)
T5 고객: "그럼 제 한도 좀 조회해 주세요"  → "한도조회"(-20) = -20
        turn_raw=-20 → turn=30. EMA: 0.6·30+0.4·48 = 37  (노랑 하단 → 곧 transfer)
        → classify: limit_inquiry → transfer_to_agent (S1 성공 경로)
```

게이지가 위험(빨강)으로 올랐다가 가치 제시 후 관심 질문→한도조회로 회복하는 전형적 S1 흐름.

## 6. 사전 운영 규칙 / Maintenance

- **추가/수정**: `CHURN-RISK-LEXICON.md`(사람용) + `churn_risk_lexicon.json`(코드용)을 **동시** 갱신 → AGENT owner PR.
  (스키마 변경 규칙은 `API.md` 헤더, `WORKFLOW.md` §3.1 준용.)
- **튜닝**: 가중치는 데모 리허설로 보정. 게이지가 한 발화에 과반응하면 `α`를 낮추거나 턴 상한을 줄입니다.
- **검증**: `hk-verify`에서 §5 예시를 골든 케이스로 사용(고정 입력 → 고정 churn_risk 궤적).

## 7. 사전 방식의 장단점 / Pros & Cons of a keyword lexicon

> 사용자 요청: "both pro and cons" — 사전 자체의 **장점/한계**를 명시.

### 장점 / Pros
- **결정적·설명 가능**: 같은 발화 → 같은 점수. `rationale`에 "어떤 키워드가 몇 점" 근거를 그대로 노출 가능.
- **저지연·저비용**: LLM 왕복 없이 STT 텍스트에서 즉시 계산 → `index_update` < 2초 목표(`PRODUCT-BRIEF.md` §5)에 유리.
- **데모 안정성**: 외부 API 실패와 무관하게 동작(장애 대응 가드레일과 정합).
- **튜닝 용이**: 가중치 표만 고치면 행동이 바뀜. 골든 테스트로 회귀 검증 가능.

### 한계 / Cons (그리고 완화책)
- **문맥·반어 약함**: "정말 좋은 조건이네요(비꼼)"를 긍정으로 오판. → §4 부정/강조 + (선택) LLM ±10 보정으로 완화.
- **표현 다양성**: 사전에 없는 신조어·사투리·완곡표현 누락. → 어간(`stems`) 부분 일치 + 리허설 기반 보강.
- **STT 오류 전파**: ko-KR 오인식 시 매칭 실패/오매칭. → 핵심 어간 위주 + 임계 근처에서만 LLM 보정.
- **가중치 임의성**: 초기 가중치는 휴리스틱(연구 기반이나 수치는 추정). → 데모 데이터로 캘리브레이션 필요(과적합 주의).
- **단일 신호 과반응**: 한 단어로 빨강 점프 위험. → EMA 평활 + 턴 상한 + 카테고리당 최대 2개 합산으로 방지.
